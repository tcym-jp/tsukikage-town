# 開発への参加

月影町の物語、操作性、セーブ互換性を守るため、変更前に[`AGENTS.md`](./AGENTS.md)と該当する仕様書を読んでください。仕様の優先順位は`PRODUCT_SPEC.md`、`STORY_BIBLE.md`、`UX_VISUAL_SPEC.md`、`ACCEPTANCE_TESTS.md`の順です。

## 開発環境

Node.js `^20.19.0`または`>=22.12.0`とnpmを用意します。Node.js 22 LTSを推奨します。

```bash
npm ci
npm run dev
```

初めてPlaywrightを使う環境では、必要に応じてChromiumを導入します。

```bash
npx playwright install chromium
```

## 変更の原則

- 進行不能、セーブ破損、モバイル操作不能を最優先で防ぎます。
- TypeScriptの`any`は原則使わず、既存の厳格設定を維持します。
- ゲーム状態・物語条件を3D表示コンポーネントへ埋め込みません。
- 会話、手紙、回覧板、イベント、クエストはデータ駆動にします。
- 既存の保存IDを安易に変更しません。互換性を変える場合はマイグレーションとテストを同じ変更へ含めます。
- プレイヤー入力をHTMLとして挿入しません。
- UI文言とゲーム本文は日本語を基本にします。
- 外部API、トラッキング、ログイン、広告、課金、生成AI APIを追加しません。
- 権利を確認できない画像、音声、3Dモデル、フォント、文章を追加しません。
- 本番デプロイ、force push、秘密情報のコミットを行いません。

## 変更の進め方

1. 変更対象に近い既存コードとテストを読みます。
2. 小さく一貫した差分で実装し、関連テストを追加または更新します。
3. `npm run lint`、`npm run typecheck`、`npm run test`、`npm run build`を通します。
4. コンテンツ変更では`npm run validate:content`と`npm run simulate:story`も通します。
5. 主要体験や見た目を変えたら`npm run test:e2e`を実行し、デスクトップとモバイルの実画面を確認します。
6. 変更理由、検証結果、既知の制約をPull Requestへ記載します。

提出前の一括確認：

```bash
npm run quality
npm run simulate:story
npm run test:e2e
npm run format:check
```

`format:check`が失敗した場合は`npm run format`を実行し、意図しない大量変更がないか差分を確認してください。

## テストの置き場所

- 純粋なゲームロジック：対象ファイルと同じ領域の`*.test.ts`
- React UI：対象コンポーネント付近の`*.test.tsx`
- コンテンツ参照と製品下限：`src/content/catalog.test.ts`
- 全日・全Endingの到達性：`src/app/simulator.test.ts`
- ユーザー操作を通した流れ：`e2e/`
- PWA公開物：`src/tests/pwa.test.ts`

バグ修正には、可能な限り修正前に失敗する回帰テストを追加します。タイマー、日付、乱数、保存時刻へ依存するテストは固定値を渡し、再現性を保ってください。テストを通すために製品要件や進行ゲートを無効化しないでください。

## UI・3D変更の確認

最低でも1366×768と390×844で確認します。影響範囲に応じて360×800、768×1024、1920×1080も確認してください。

- 日本語の文字切れ、重なり、長文スクロール
- キーボードだけでの操作、フォーカスリング、Escape、ダイアログのフォーカストラップ
- 44px程度のタッチ領域、safe-area、縦向きと横向き
- 200%ズーム、文字サイズ「最大」、モーション軽減、ミュート
- 夕方、夜、低画質、WebGLエラー案内
- 会話・メニュー中に町が誤移動しないこと
- コンソールエラーと未処理Promise rejectionがないこと

手動確認用スクリーンショットは`artifacts/screenshots/desktop/`と`artifacts/screenshots/mobile/`へ保存します。本リポジトリでは完成判定の証跡として主要24枚をGit管理します。中間確認画像を追加した場合は、最終成果物と区別できる名前にするか、コミット前に整理してください。

## コンテンツ変更

人物、手紙、回覧板、手掛かり、会話、章を変更する場合は[`CONTENT_GUIDE.md`](./CONTENT_GUIDE.md)に従います。特に次を確認してください。

- 新しいIDを`src/content/contentIds.ts`へ集中定義したか
- 重要情報へ2媒体以上の取得経路があるか
- 会話ノードがすべて入口から到達可能か
- `STORY_ACTIONS`と`STORY_BEATS`へプレイ可能な接続があるか
- Ending A/B/Cの到達性とDのフォールバックを壊していないか
- 古いセーブで不明IDが発生しないか

物語本文の大きな変更は、`STORY_BIBLE.md`の人物動機、真相、文章規則との整合も説明してください。

## アセットと音

低ポリゴン形状、SVG、Canvas、CSS、Web Audio APIなど、リポジトリ内で生成・編集できる素材を優先します。外部素材を使う場合は、作者、取得元、ライセンス、変更、用途を[`THIRD_PARTY_NOTICES.md`](./THIRD_PARTY_NOTICES.md)へ記録します。

アイコンとOG画像は次のコマンドで再生成できます。

```bash
npm run assets:generate
```

このコマンドは`public/icons/`と`public/og.png`を上書きします。実行後は画像サイズ、透過、マスカブルアイコンの安全領域、日本語表記を目視確認してください。

## Pull Requestのチェックリスト

- [ ] 変更の目的とユーザーへの影響を説明した
- [ ] 無関係なファイルや既存変更を上書きしていない
- [ ] lint、型検査、単体・コンポーネントテスト、ビルドが成功した
- [ ] コンテンツ変更では整合性検証と物語シミュレーションが成功した
- [ ] 主要体験変更ではデスクトップ・モバイルE2Eを確認した
- [ ] 見た目変更では実画面とスクリーンショットを確認した
- [ ] セーブ形式・ID変更にはマイグレーションと回帰テストがある
- [ ] アクセシビリティ、モーション軽減、音なしの情報伝達を確認した
- [ ] 新しい依存・素材のライセンスを記録した
- [ ] 秘密情報、実在個人のデータ、生成物の不要ファイルを含めていない
